Azure Synapse Analytics

Synapse Analytics is where your cleaned, processed data gets exposed for consumption. It is the SQL serving layer — analysts query Gold data here using familiar SQL, and Power BI connects to it for dashboards and reports.

What Synapse is and where it fits

By the time data reaches Azure Synapse, your data engineering pipeline has already done the hard work. Raw data was ingested by ADF, cleaned by Databricks, and stored in Gold Delta tables in ADLS Gen2. Synapse's job is to make that Gold data accessible to SQL-speaking consumers — analysts, Power BI, reporting tools — without them needing to know anything about Spark, Delta Lake, or data lakes.

The most common pattern is using Synapse's Serverless SQL Pool to create external tables that point directly at the Delta files in ADLS Gen2. Analysts run standard SQL queries. Synapse handles reading the Parquet files underneath — no data duplication, no ETL into a separate warehouse.

The three types of Synapse pools

Serverless SQL PoolMost common for data engineers

Query data directly in ADLS Gen2 using SQL — no database to provision. Pay per query. Perfect for ad-hoc exploration and external tables over Delta Lake. This is what most data engineering teams use for serving Gold layer data.

Dedicated SQL PoolFor high-concurrency BI workloads

A traditional data warehouse with reserved compute capacity. High performance for heavy, concurrent query workloads. More expensive — you pay even when idle. Use when you have many analysts running complex queries simultaneously.

Apache Spark PoolAlternative to Databricks

Run PySpark notebooks directly inside Synapse, without switching to Databricks. Good for teams that want one unified platform. In practice, many teams still prefer standalone Databricks for transformation work.

🎯 Pro Tip
For most Azure data engineering projects, Serverless SQL Pool is all you need. It queries ADLS Gen2 directly, costs nothing when idle, and connects seamlessly to Power BI. Dedicated SQL Pool is only worth the cost if you have dozens of analysts running heavy queries simultaneously.

Creating external tables over Delta Lake

This is the most important Synapse skill for data engineers. You create an external table in Synapse that points to your Gold Delta files in ADLS Gen2. Analysts can then query it like a regular SQL table. When the Databricks pipeline updates the Gold data, the Synapse external table immediately reflects those changes — no sync job needed.

create_external_table.sql
sql
-- Create an external table in Synapse that points to Delta Lake in ADLS Gen2
-- Analysts query this like a normal SQL table — no data movement needed

-- First create the external data source
CREATE EXTERNAL DATA SOURCE GoldLayer
WITH (
    LOCATION = 'abfss://gold@yourlake.dfs.core.windows.net/',
    CREDENTIAL = [ManagedIdentityCredential]
);

-- Create external file format for Delta/Parquet
CREATE EXTERNAL FILE FORMAT DeltaFormat
WITH (FORMAT_TYPE = DELTA);

-- Create the external table — analysts query this directly
CREATE EXTERNAL TABLE dbo.daily_sales_summary (
    order_date      DATE,
    region          NVARCHAR(100),
    product_category NVARCHAR(200),
    total_revenue   FLOAT,
    order_count     INT,
    avg_order_value FLOAT,
    unique_customers INT,
    units_sold      INT
)
WITH (
    LOCATION = 'daily_sales_summary/',
    DATA_SOURCE = GoldLayer,
    FILE_FORMAT = DeltaFormat
);

Writing analytics queries in Synapse

Once your external tables are in place, analysts write standard T-SQL queries. Synapse supports window functions, CTEs, and all the advanced SQL features you learned in the SQL for Data Engineers module.

analytics_queries.sql
sql
-- Real analytics queries that run on Synapse over the Gold layer

-- 1. Top performing regions this month
SELECT
    region,
    SUM(total_revenue)    AS monthly_revenue,
    SUM(order_count)      AS monthly_orders,
    AVG(avg_order_value)  AS avg_basket_size,
    RANK() OVER (ORDER BY SUM(total_revenue) DESC) AS revenue_rank
FROM dbo.daily_sales_summary
WHERE order_date >= DATEADD(month, DATEDIFF(month, 0, GETDATE()), 0)
GROUP BY region
ORDER BY monthly_revenue DESC;

-- 2. Week-over-week revenue growth by category
WITH weekly AS (
    SELECT
        DATEPART(week, order_date) AS week_num,
        product_category,
        SUM(total_revenue) AS weekly_revenue
    FROM dbo.daily_sales_summary
    GROUP BY DATEPART(week, order_date), product_category
)
SELECT
    week_num,
    product_category,
    weekly_revenue,
    LAG(weekly_revenue) OVER (PARTITION BY product_category ORDER BY week_num) AS prev_week_revenue,
    ROUND(
        (weekly_revenue - LAG(weekly_revenue) OVER (PARTITION BY product_category ORDER BY week_num))
        / NULLIF(LAG(weekly_revenue) OVER (PARTITION BY product_category ORDER BY week_num), 0) * 100,
        2
    ) AS wow_growth_pct
FROM weekly
ORDER BY week_num DESC, weekly_revenue DESC;

Access control — who sees what

In production, you always control who can access which data. Synapse uses standard SQL roles and permissions. The data engineering team gets write access to manage schemas. Analysts get read-only access to the Gold schema only. They never touch Bronze or Silver directly.

rbac_setup.sql
sql
-- Synapse RBAC — control who can access what
-- Run these in Synapse SQL pool as an admin

-- Create a role for the analytics team (read-only on Gold)
CREATE ROLE analytics_reader;
GRANT SELECT ON SCHEMA::dbo TO analytics_reader;

-- Create a role for data engineers (can create/modify objects)
CREATE ROLE data_engineer;
GRANT CONTROL ON SCHEMA::dbo TO data_engineer;

-- Add users to roles
EXEC sp_addrolemember 'analytics_reader', 'analyst@yourcompany.com';
EXEC sp_addrolemember 'data_engineer',    'engineer@yourcompany.com';
☁️ Azure
Synapse connects natively to Power BI through the Azure Synapse Link for Power BI feature. Once connected, Power BI report authors see the Synapse tables in the Power BI Desktop data source picker — no manual connection strings needed.

🎯 Key Takeaways

  • ✓Synapse Analytics is the SQL serving layer — it exposes your Gold Delta data to analysts and Power BI using standard SQL
  • ✓Serverless SQL Pool queries ADLS Gen2 directly — no data movement, pay per query, zero idle cost
  • ✓External tables over Delta Lake are the key pattern — create them once, analysts query them like normal tables
  • ✓Dedicated SQL Pool is for high-concurrency workloads with many analysts — overkill for most starting projects
  • ✓Always use RBAC — analysts get SELECT on the Gold schema only, never Bronze or Silver
  • ✓Synapse connects natively to Power BI — no manual connection configuration needed
  • ✓When Databricks updates the Gold Delta files, Synapse external tables reflect those changes immediately
