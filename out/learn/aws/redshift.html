Amazon Redshift

Amazon Redshift is AWS's cloud data warehouse — the SQL serving layer where analysts run queries against large, processed datasets. It uses columnar storage and massively parallel processing to handle complex analytics queries at scale.

Redshift vs. Azure Synapse

Amazon Redshift is AWS's answer to Azure Synapse Analytics Dedicated SQL Pool. Both are column-oriented data warehouses designed for analytics workloads — fast aggregations, complex joins, large GROUP BY operations. Both charge for provisioned compute capacity whether it's being used or not.

The key Redshift concept that has no Azure equivalent is distribution keys. Because Redshift physically distributes data across multiple nodes in a cluster, how you choose to distribute rows across those nodes dramatically affects query performance. Choosing the right DISTKEY for each table is one of the most important Redshift DBA skills.

Columnar Storage

Data stored column-by-column, not row-by-row. When you query SUM(revenue), Redshift reads only the revenue column — not the entire row. Dramatically faster for analytics.

MPP Architecture

Massively Parallel Processing. A Redshift cluster has a leader node and multiple compute nodes. Queries are split and run in parallel across all nodes simultaneously.

Redshift Spectrum

Query S3 data directly without loading it into Redshift. Combines the speed of Redshift with the scale of S3 — query petabytes of data without storage costs.

Loading data into Redshift

The COPY command is the fastest and recommended way to load data from S3 into Redshift. It leverages Redshift's parallel nodes to read from S3 simultaneously — much faster than inserting row by row.

redshift_load.sql
sql
-- Load data from S3 into Redshift using COPY command
-- This is the fastest way to bulk-load data into Redshift

COPY sales_gold
FROM 's3://yourcompany-datalake-gold/daily_sales_summary/'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftS3Role'
FORMAT AS PARQUET;

-- For incremental loads — use a staging table + MERGE pattern
CREATE TEMP TABLE sales_staging (LIKE sales_gold);

COPY sales_staging
FROM 's3://yourcompany-datalake-gold/daily_sales_summary/2025/03/01/'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftS3Role'
FORMAT AS PARQUET;

-- Upsert: update existing records, insert new ones
MERGE INTO sales_gold
USING sales_staging AS src
ON sales_gold.order_id = src.order_id
WHEN MATCHED THEN UPDATE SET
    total_revenue = src.total_revenue,
    order_count   = src.order_count
WHEN NOT MATCHED THEN INSERT VALUES (src.*);

DROP TABLE sales_staging;

Distribution keys and sort keys

This is the most important Redshift performance concept. Setting the right DISTKEY (how rows are distributed across nodes) and SORTKEY (how data is physically ordered on disk) for each table can make queries 10–100x faster.

redshift_table_design.sql
sql
-- Redshift distribution and sort keys — critical for performance
-- Set these when creating tables, not after

CREATE TABLE sales_gold (
    order_date      DATE        NOT NULL,
    region          VARCHAR(100),
    product_category VARCHAR(200),
    total_revenue   DECIMAL(15,2),
    order_count     INTEGER,
    customer_id     BIGINT
)
DISTSTYLE KEY             -- Distribute rows by this column across nodes
DISTKEY (region)          -- Rows with same region go to same node (good for region-based joins)
SORTKEY (order_date);     -- Physically sort data by date (speeds up date range queries)

-- ANALYZE and VACUUM — keep statistics and storage healthy
ANALYZE sales_gold;            -- Update query optimizer statistics
VACUUM SORT ONLY sales_gold;   -- Re-sort after inserts without reclaiming space (faster)
VACUUM sales_gold;             -- Full vacuum: reclaim space + re-sort (slower, run weekly)

Redshift Spectrum — query S3 without loading

Redshift Spectrum lets you query data sitting in S3 using standard SQL, without loading it into Redshift first. It reads from the Glue Data Catalog for table schemas. This is perfect for querying historical data that you don't want to pay to store in Redshift.

redshift_spectrum.sql
sql
-- Redshift Spectrum — query S3 directly without loading data
-- Creates an external schema pointing to the Glue Data Catalog

CREATE EXTERNAL SCHEMA spectrum_gold
FROM DATA CATALOG
DATABASE 'gold_db'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftSpectrumRole'
CREATE EXTERNAL DATABASE IF NOT EXISTS;

-- Now query S3 data as if it were a Redshift table
-- No data movement — Spectrum reads Parquet directly from S3
SELECT
    region,
    SUM(total_revenue) AS total_rev,
    COUNT(*) AS order_count
FROM spectrum_gold.daily_sales_summary
WHERE order_date >= '2025-01-01'
GROUP BY region
ORDER BY total_rev DESC;
🎯 Pro Tip
A common cost-saving pattern: keep only the last 90 days of data in Redshift (fast, expensive), and use Redshift Spectrum to query older data directly in S3 (slower, cheap). Analysts get fast queries for recent data and can still access historical data when needed.

🎯 Key Takeaways

  • ✓Redshift is AWS's data warehouse — equivalent to Azure Synapse Dedicated Pool. Columnar, MPP, built for analytics
  • ✓The COPY command is the correct way to bulk-load data from S3 — never use row-by-row INSERT for large datasets
  • ✓DISTKEY and SORTKEY are critical for performance — choose them based on your most common query patterns
  • ✓DISTSTYLE KEY distributes rows by a column — put the column you most often JOIN or GROUP BY as the DISTKEY
  • ✓VACUUM and ANALYZE regularly — without them, query performance degrades as data accumulates and statistics become stale
  • ✓Redshift Spectrum queries S3 directly without loading — great for cold/historical data you want to query but not pay to store in Redshift
