Amazon Kinesis — Real-Time Streaming

Amazon Kinesis is AWS's managed real-time data streaming service. Equivalent to Azure Event Hubs, it captures millions of events per second from websites, mobile apps, and IoT devices — the foundation of every real-time AWS data pipeline.

The four Kinesis services

"Kinesis" is actually a family of four related services, not one. Understanding which one to use for which purpose is important — they serve different parts of the streaming pipeline.

Core streaming backbone
Kinesis Data Streams

The raw event stream. Producers write events in, consumers read events out. Low latency (milliseconds). You manage shards (capacity units) manually. Most flexible but requires consumer code.

Stream → S3 delivery
Kinesis Data Firehose

Fully managed delivery service. Buffers incoming events and automatically writes them to S3, Redshift, or OpenSearch. No consumer code needed. Perfect for landing streaming data in your Bronze layer.

Real-time SQL on streams
Kinesis Data Analytics

Run SQL queries directly against a live data stream. Equivalent to Azure Stream Analytics. Use for real-time aggregations, anomaly detection, and filtering — without writing a consumer app.

Video/media streaming
Kinesis Video Streams

Specialized service for streaming video from cameras and IoT devices. Not typically used in data engineering — included for completeness.

Writing events to Kinesis (Producer)

Any application can write events to a Kinesis Data Stream using the AWS SDK. The partition key determines which shard an event goes to — events with the same partition key always go to the same shard, preserving ordering for that key.

kinesis_producer.py
python
# Kinesis Producer — send real-time events to a Kinesis Data Stream
# Example: website clickstream events sent as they happen

import boto3
import json
from datetime import datetime
import random

kinesis = boto3.client('kinesis', region_name='us-east-1')
STREAM_NAME = 'clickstream-events'

def send_click_event(user_id: str, page: str, action: str):
    event = {
        "event_id":  f"evt_{random.randint(100000, 999999)}",
        "user_id":   user_id,
        "page":      page,
        "action":    action,
        "timestamp": datetime.utcnow().isoformat(),
        "session_id": f"sess_{user_id[:8]}"
    }
    
    kinesis.put_record(
        StreamName=STREAM_NAME,
        Data=json.dumps(event),
        PartitionKey=user_id  # Same user's events always go to same shard
    )

# Send a batch of events (more efficient than one at a time)
def send_batch(events: list):
    records = [
        {"Data": json.dumps(e), "PartitionKey": e["user_id"]}
        for e in events
    ]
    kinesis.put_records(StreamName=STREAM_NAME, Records=records)

# Simulate 100 click events
for i in range(100):
    send_click_event(f"user_{i}", "/products/laptop", "view")

print("✅ 100 click events sent to Kinesis")

Kinesis Firehose — auto-land events in S3

For most data engineering use cases, you don't want to write a consumer application. You just want streaming events to land in your Bronze S3 layer automatically. Kinesis Firehose does exactly that — it buffers events from a Data Stream and delivers them to S3 in batches, converting to Parquet automatically.

firehose_setup.py
python
# Kinesis Data Firehose — auto-delivers stream to S3
# No consumer code needed — Firehose buffers and writes to S3 automatically

import boto3

firehose = boto3.client('firehose', region_name='us-east-1')

# Create a Firehose delivery stream to S3
firehose.create_delivery_stream(
    DeliveryStreamName='clickstream-to-s3',
    DeliveryStreamType='KinesisStreamAsSource',
    KinesisStreamSourceConfiguration={
        'KinesisStreamARN': 'arn:aws:kinesis:us-east-1:123456789:stream/clickstream-events',
        'RoleARN': 'arn:aws:iam::123456789:role/FirehoseRole'
    },
    ExtendedS3DestinationConfiguration={
        'RoleARN': 'arn:aws:iam::123456789:role/FirehoseRole',
        'BucketARN': 'arn:aws:s3:::yourcompany-datalake-bronze',
        'Prefix': 'clickstream/!{timestamp:yyyy}/!{timestamp:MM}/!{timestamp:dd}/',
        'BufferingHints': {
            'SizeInMBs': 128,       # Buffer 128MB before writing
            'IntervalInSeconds': 300 # Or write every 5 minutes, whichever comes first
        },
        'CompressionFormat': 'GZIP',
        'DataFormatConversionConfiguration': {
            'Enabled': True,
            'OutputFormatConfiguration': { 'Serializer': { 'ParquetSerDe': {} } }
        }
    }
)
print("✅ Firehose delivery stream created — events will flow to S3 automatically")
🎯 Pro Tip
The most common real-world pattern is: Kinesis Data Streams (capture events) → Kinesis Firehose (buffer and deliver to S3) → AWS Glue (process the landed files) → Redshift (serve analytics). You'll see this exact pattern in almost every AWS streaming architecture interview question.

Reading events from Kinesis (Consumer)

kinesis_consumer.py
python
# Kinesis Consumer — read and process events in real time
# Uses Kinesis Client Library (KCL) pattern

import boto3
import json
import time

kinesis = boto3.client('kinesis', region_name='us-east-1')
STREAM_NAME = 'clickstream-events'

# Get all shards in the stream
shards = kinesis.list_shards(StreamName=STREAM_NAME)['Shards']

for shard in shards:
    # Get iterator starting from the latest record
    iterator_response = kinesis.get_shard_iterator(
        StreamName=STREAM_NAME,
        ShardId=shard['ShardId'],
        ShardIteratorType='LATEST'  # LATEST, TRIM_HORIZON, AT_TIMESTAMP
    )
    
    shard_iterator = iterator_response['ShardIterator']
    
    # Poll for new records
    while True:
        response = kinesis.get_records(ShardIterator=shard_iterator, Limit=100)
        
        for record in response['Records']:
            event = json.loads(record['Data'])
            
            # Process each event
            print(f"User {event['user_id']} viewed {event['page']} at {event['timestamp']}")
        
        shard_iterator = response['NextShardIterator']
        time.sleep(1)  # Poll every second

🎯 Key Takeaways

  • ✓Kinesis is a family of four services — Data Streams (raw stream), Firehose (delivery), Analytics (SQL), Video (media)
  • ✓Kinesis Data Streams is the core backbone — producers write in, consumers read out, millisecond latency
  • ✓Kinesis Firehose automatically delivers stream data to S3 — no consumer code needed, converts to Parquet
  • ✓The partition key determines which shard an event goes to — use user_id or entity_id to preserve ordering per entity
  • ✓Kinesis Data Analytics runs SQL directly on live streams — equivalent to Azure Stream Analytics
  • ✓Most common pattern: Kinesis Streams → Firehose → S3 Bronze → Glue → Redshift
  • ✓Shards are the capacity unit — each shard handles 1MB/sec write and 2MB/sec read. Add shards as throughput grows
